# eGovFrame Strategy

## Strategic Position

Agent Estate should start as an independent open-source VS Code extension that supports eGovFrame 5.0 projects.

Do not start by asking eGovFrame to accept a large runtime feature. First build evidence that the extension is useful.

## Why eGovFrame Fits

The 2026 eGovFrame contribution notice includes an idea category related to AI agents, LLM integration, and MCP proposals. That makes the topic highly relevant, but the timing should be controlled.

The right strategy is:

```text
internal MVP
-> open-source release
-> eGovFrame 5.0 sample/docs
-> contribution participation
-> official repo reference or compatibility confirmation review
```

## Officialness Levels

### 1. Contribution Recognition

This may include contribution participation, awards, appreciation letters, or portal records.

Useful for reference, but not the same as product certification.

### 2. Official Repository Inclusion

If a guide, sample, or small extension-related contribution lands in an eGovFrame official repository, that becomes a stronger trust signal.

Possible targets:

- `egovframe-docs`
- `egovframe-ai-rag`
- `egovframe-vscode-initializr`

### 3. Compatibility Confirmation

Formal compatibility confirmation is a separate process. It may be relevant later if Agent Estate becomes a mature support tool for eGovFrame development environments.

The likely category to investigate is support software such as Case Tool or ALM-style development support.

## Recommended Contribution Timing

Do not submit immediately.

Submit only after:

- a local scan works
- a report can be exported
- a VS Code command or panel exists
- the eGovFrame 5.0 workflow is documented
- the project has a clean open-source README and license direction

## First Public Message

When ready, the external framing should be:

`AI Agent & MCP Governance Extension for eGovFrame development environments`

Avoid claiming:

- official eGovFrame certification
- replacement for endpoint security
- automatic compliance
